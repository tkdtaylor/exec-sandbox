## Core behaviors

### B-001: Run a payload in a no-network sandbox

- **Trigger:** `exec-sandbox run` receives a JSON `RunRequest` on stdin with a `run.payload`.
- **Response:** writes `req.run.payload` to a `payload.sh` (mode 0600) in a fresh temp dir, selects the isolation backend by `run.tier` (see B-008), then runs the payload under that backend with a minimal read-only root (`/usr`, `/etc`, conditionally `/bin /lib /lib64 /sbin`), `/proc`, `/dev`, a `/tmp` tmpfs, `PATH=/usr/bin:/bin`, the payload bind-mounted read-only as `/payload.sh`, and the egress proxy socket bind-mounted as `/proxy.sock` — with **no network namespace regardless of tier**. When `run.workdir` is set, the named host directory is additionally bind-mounted **read-write** at `/work` and becomes the payload's cwd (see B-010); otherwise there is no `/work` mount. For the bubblewrap tier this is `bwrap --seccomp <fd> --unshare-all --die-with-parent --clearenv` — the `--seccomp` flag installs a default-deny syscall profile (see B-013); for the gvisor tier it is `runsc run` over a generated OCI bundle whose spec declares an empty network namespace (see B-008). Captures stdout, stderr, and exit code. Returns a JSON result `{stdout, stderr, exit_code, sandbox_status}` on stdout.
- **Side effects:** the temp dir and its contents (payload, proxy socket; for gvisor also the OCI bundle in its own temp dir) are created and removed (the one-shot `baseline.teardown()` → `os.RemoveAll(work)` + `proxy.Wipe()`, plus the backend cleanup func); spawn and exit audit events are emitted (see B-004).
- **Failure modes:** if the selected runtime (`bwrap` or `runsc`) fails to start (e.g. binary absent or non-exec error), `exit_code` is set to `127` and the error string is appended to stderr. If the runtime exits non-zero, that exit code is propagated. An unrecognized tier returns `{error: "tier not implemented: <tier>"}` without running anything (see B-008).
- **References:** ADR-001 D2/D3, ADR-002; `run.go` `Run` / `backendFor` / `bwrapArgv`; `gvisor.go`.

### B-002: Enforce the egress allowlist and route through the proxy

- **Trigger:** the sandboxed payload makes an HTTP request through `/proxy.sock`.
- **Response:** the egress proxy strips the port from the request Host and checks it against the allowlist derived from `profile.capabilities[NetConnect].allowlist` (**host check, first**). If the host is allowlisted, the proxy then checks the request's HTTP method (normalized to canonical upper-case) against that host's optional verb set from `profile.capabilities[NetConnect].methods` (**verb check, second** — ADR 008): a host with no/empty verb set is unconstrained (all verbs pass); a host with a non-empty set permits only those verbs. A request that passes both checks is resolved to a real origin via the `origin_map` (`host -> {ip, port}`) and forwarded; the upstream status and body are returned. The sandbox itself has no network namespace, so the proxy socket is the only way out. The verb check only **narrows** egress — it never opens a route the host check would have denied, and a blocked verb produces **no** outbound connection and **no** credential injection.
- **Side effects:** an outbound HTTP request to the real origin (with a credential header injected if one was loaded for that host — see B-003). A blocked host or blocked verb produces no outbound request at all.
- **Failure modes:** a non-allowlisted host returns `403 blocked-by-allowlist`. An allowlisted host whose verb set excludes the request's method returns `403 blocked-by-method` (same status, distinct body) with no upstream connection and no credential injected. An allowlisted, verb-permitted host with no `origin_map` entry returns `502 no-route`. An upstream/dial error returns `502` with the error string.
- **References:** ADR-001 D3/D4, ADR 008 (per-host verb allowlist; decide=policy-engine / enforce=here); `proxy.go` `handle`, `run.go` `netVerbAllowlist`; tests `TestSandboxReachesAllowlistedHostViaProxy`, `TestProxyBlocksNonAllowlistedHost`, `TestProxyForwardsAllowedVerb`, `TestProxyBlocksDisallowedVerb`, `TestProxyHostCheckPrecedesVerbCheck`, `TestSandboxDisallowedVerbBlockedOriginSeesNothing`.

### B-003: Inject vault credentials at spawn (proxy mode never in sandbox; env mode delivered + wiped)

- **Trigger:** `req.run.secret_refs` contains one or more opaque handles.
- **Response:** for each handle, exec-sandbox mints a `sandbox_identity` and calls `vault.inject(handle, sandbox_identity, mode)` over the `vault_socket`. The identity shape depends on `wiring.attestation_key` (ADR 017): set → a host-signed `{sandbox_id, tier, profile_digest, created_at, nonce, attestation_format:"host-ed25519/v2", attestation}` (no pubkey; vault verifies the ed25519 signature against the operator-published trust root over the v2 preimage); unset → the transitional ADR 014 self-attestation `{sandbox_id, nonce, ts, attestation_pubkey, attestation}`. A configured-but-broken key aborts the run with `{error}` before this loop (see the mint step below). The response's `delivery` field selects how the credential is handled, and the two modes are deliberately distinct:
  - **proxy mode** (`delivery == "proxy"`): exec-sandbox loads the returned credential onto the egress proxy for the binding's host (`Header: Scheme Value`, defaulting to `Authorization: Bearer`). The credential value is **never** placed into the sandbox env, args, stdout, or payload — it lives only on the host-side proxy and is injected into allowlisted outbound requests to that host. This is the F-002 invariant and it is unchanged.
  - **env mode** (`delivery == "env"`): exec-sandbox **deliberately delivers** the credential value into the sandbox process environment under the vault-specified `var_name` (a documented exception to the proxy-mode rule — some tools read a token from `$VAR` rather than via an injected header). The value reaches the sandbox **off the spawn argv** (Tier-1 bwrap via `--args FD`; Tier-2 gVisor via the OCI `process.env` file), so it never appears in `/proc/<pid>/cmdline`. The host-side copy is held in one place and **wiped per the wipe clock**: immediately after the child is spawned (post-spawn) and again at teardown — no host-side copy survives the run. A response missing `var_name` (or otherwise malformed) is treated as an inject failure (see below) — no empty/unnamed var is delivered.
- **Side effects:** a `vault.inject` IPC call per handle; `secrets_injected` entries (`{handle_prefix, delivery}`, 8-char prefix, never the full handle, never the value) recorded in `sandbox_status`; in proxy mode the proxy's credential map is populated and wiped at teardown; in env mode the host-side env-credential holder is populated and wiped post-spawn + at teardown. A mixed run (some handles proxy, some env) handles each independently. The credential value never appears in the returned `result`, `sandbox_status`, or any audit event for either mode.
- **Failure modes:** if the inject call errors, returns an `error` field, or (env mode) omits `var_name`, exec-sandbox emits an `inject_failed` audit event (decision `deny`) for that handle and continues with the remaining handles (the run is not aborted); no partial env var is delivered for a skipped env-mode handle.
- **References:** ADR-001 D5; ADR-015 (env-mode delivery + wipe clock); `run.go` `Run` inject loop / `vaultInject`; `proxy.go` `SetCredential` / `Wipe`; `envcreds.go` `EnvCredentials` / `Wipe`.

### B-004: Emit spawn / inject_failed / exit audit events

- **Trigger:** the lifecycle of a run — at spawn, on a failed injection, and at exit.
- **Response:** emits JSON-lines audit events to the `audit_socket`. `spawn` (`decision: allow`, context `{tier, request_id}`) is emitted first — before the baseline snapshot (B-012) and the inject loop; `inject_failed` (`decision: deny`) per failed handle; `exit` (`decision: allow`, context `{exit_code, duration_ms, request_id}`) after the payload finishes.
- **Side effects:** one `emit` IPC call per event.
- **Failure modes:** emission is best-effort. An empty `audit_socket` makes `emit` a no-op; an IPC error is swallowed (the run is not aborted).
- **References:** ADR-001 D6; `run.go` `emit`.

### B-005: Report sandbox status in the result

- **Trigger:** a run completes (any exit code).
- **Response:** the result's `sandbox_status` carries `{sandbox_id, tier, duration_ms, secrets_injected, status, limits}`. `status` is `"clean"` normally, or `"timeout"` when the payload was killed by the `timeout_sec` wall-clock deadline (see B-009). `tier` echoes `req.run.tier`. `secrets_injected` lists `{handle_prefix, delivery}` per successfully injected handle (handles are truncated to an 8-char prefix — never the full handle, never the credential). `limits` is the applied-caps record `{cpu_count, memory_mb, pids, disk_mb, timeout_sec, max_output_bytes, degraded[], output_truncated[]}` (zeros mean "no limit requested"; `degraded` lists any secondary cap the host could not enforce; `output_truncated` lists the streams whose output cap dropped bytes — `[]`, `["stdout"]`, or `["stdout","stderr"]` — see B-009).
- **Side effects:** none beyond the returned JSON.
- **Failure modes:** none specific; this is assembled unconditionally.
- **References:** `run.go` `Run` return / `limitsReport`; [data-model.md](data-model.md).

### B-008: Select the isolation backend by tier

- **Trigger:** a run reaches the spawn step with `req.run.tier`.
- **Response:** `backendFor(tier)` selects the isolation backend. `""` and `"bubblewrap"` select the Tier-1 bubblewrap backend (`bwrapArgv`), which additionally installs a default-deny seccomp profile via `--seccomp <fd>` (see B-013); `"gvisor"` selects the Tier-2 gVisor backend; `"firecracker"` selects the Tier-3 Firecracker microVM backend (see B-015). The gVisor backend writes an OCI bundle (`config.json` + a rootfs dir) to its own temp dir: the spec declares a `network` namespace with **no path** (a fresh empty netns — loopback only, no host/bridged networking, the OCI equivalent of `--unshare-all`), a read-only root with the host system dirs bind-mounted read-only, the payload read-only at `/payload.sh`, and the proxy socket as the only egress bind-mount at `/proxy.sock`. It then runs `runsc --network=none --host-uds=open --ignore-cgroups run --bundle <dir> <id>` (and `--rootless` when exec-sandbox runs unprivileged). `--host-uds=open` lets the payload connect to the existing proxy socket but never create host sockets. The bundle is removed after the process exits.
- **Side effects:** for the gVisor tier, an OCI bundle temp dir is created and removed (backend cleanup func); for the Firecracker tier, a per-run bundle (config + payload drive) and a vsock bridge are created and torn down (see B-015). The chosen backend's runtime binary (`bwrap`, `runsc`, or `bwrap`-wrapping-`firecracker`) is exec'd.
- **Failure modes:** any tier other than `""`/`bubblewrap`/`gvisor`/`firecracker` returns `{error: "tier not implemented: <tier>"}` — there is **no silent fall-back** to bubblewrap, and no payload runs. A bundle-write error returns `{error: <err>}`. For the Firecracker tier, a missing/unverifiable pinned kernel or rootfs, an absent `firecracker` binary, or an inaccessible `/dev/kvm` is a spawn error (`exit_code 127`), never a fall-back (see B-015).
- **References:** ADR-001 D7, ADR-002, ADR-010; `run.go` `backendFor` / `Backend` / `bubblewrapBackend`; `gvisor.go` `gvisorBackend` / `gvisorOCISpec`; `firecracker.go` `firecrackerBackend`; tests `TestBackendForRoutesByTier`, `TestBackendForUnknownTierErrors`, `TestFirecrackerBackendForReturnsBackend`, `TestGvisorSpecHasNoSharedNetwork`, `TestGvisorSpecMountsOnlyProxySocketForEgress`, `TestGvisorRunReachesAllowlistedHostAndBlocksOthers`.

### B-009: Enforce profile.limits (cpu / memory / pids / disk / wall-clock / output)

- **Trigger:** a run carries a `run.profile.limits` object with one or more of `cpu_count`, `memory_mb`, `pids`, `disk_mb`, `timeout_sec`, `max_output_bytes` (parsed by `parseLimits`; a missing/zero/non-positive field means "no limit").
- **Response:** each cap is enforced on the selected backend (ADR 003; the Firecracker mapping is ADR 010 D4):
  - `timeout_sec` — backend-agnostic, in `Run()`: the child runs in its own process group (`Setpgid`) under a `context.WithTimeout`; on the deadline the whole group is `SIGKILL`ed, `sandbox_status.status` becomes `"timeout"`, and `exit_code` is `137`. **Not** mapped into the Firecracker config (host-side, above the seam).
  - `memory_mb` → `RLIMIT_AS`: under bubblewrap via an in-sandbox `prlimit --as`; under gVisor via OCI `process.rlimits` (sentry-enforced). **Under Firecracker → `machine-config.mem_size_mib`** — the microVM's hard RAM ceiling; a payload that exceeds it is **OOM-killed by the guest kernel** (`exit_code 137`, `"Killed"` on stdout), whereas under the namespace tiers the allocator kills it.
  - `pids` → `RLIMIT_NPROC`: under bubblewrap via in-sandbox `prlimit --nproc` (per-sandbox because the bwrap user namespace gives a fresh process count); under gVisor via OCI `process.rlimits`. **Under Firecracker → an in-guest `RLIMIT_NPROC`**: the host emits `exec_sandbox.nproc=N` on the kernel cmdline (only when `pids > 0`), and the guest `/sbin/init` applies `ulimit -u N` **and drops the payload to nobody** (`/usr/bin/setpriv --reuid 65534 --regid 65534 --clear-groups`) before running it — the privilege drop is load-bearing because the kernel does **not** enforce `RLIMIT_NPROC` for a uid-0 process (the guest owns its pid space, the analogue of the bwrap userns). A fork bomb hits the cap ("Cannot fork" / "can't fork: Resource temporarily unavailable").
  - `disk_mb` → writable-layer size cap: bubblewrap `--size <bytes> --tmpfs /tmp`; gVisor `/tmp` tmpfs `size=` mount option. **Under Firecracker → the size of the writable payload drive (vdb)** presented to the guest. A write past the cap returns ENOSPC.
  - `cpu_count` → under bubblewrap/gVisor a `taskset -c 0-(N-1)` affinity prefix on the spawn argv (inherited into the sandbox; visible in-box under bubblewrap via `nproc`, verified host-side by the argv record under gVisor — ADR 003 / agent-builder ADR 028). **Under Firecracker → `machine-config.vcpu_count`** — the microVM literally has N vCPUs, a **real** cap (the guest reports `nproc == N`) with **no** `taskset` prefix on the argv; this is a *stronger* enforcement than the namespace tiers' affinity hint, recorded as such and never a degrade (ADR 010 D4).
  - `max_output_bytes` → host-side capture cap **above** the `tier` seam (ADR 007): `Run()` captures each of stdout/stderr through a capping writer that retains at most `max_output_bytes` bytes per stream and **drops** the overflow, while reporting every write as fully consumed so the payload's pipe never sees a short write or error (the cap is a host memory guard, not a payload signal — the payload runs to its natural completion and keeps its own exit code). stdout and stderr are capped **independently** at the same ceiling. Because the cap lives in the host capture path, the bwrap argv and the OCI spec are **unchanged** by it and the truncated length + `output_truncated` record are **identical** under bubblewrap and gVisor (tier-independent). Writing exactly the cap does not flag truncation; the cap+1th byte does. A run with no `max_output_bytes` (or 0) captures full output and records `output_truncated: []` (byte-for-byte the prior behavior).
- **Side effects:** the spawn argv and/or OCI `config.json` carry the caps; under Firecracker the **machine-config** carries `vcpu_count`/`mem_size_mib`, the **writable payload drive** is sized by `disk_mb`, and the **kernel cmdline** carries `exec_sandbox.nproc=N` (`pids`). In all cases `max_output_bytes` and `timeout_sec` are host-side and touch none of these — the Firecracker config is **byte-for-byte independent** of both. `sandbox_status.limits` records the applied values (including `max_output_bytes` and the `output_truncated` list of capped streams); the `exit` audit context carries `status`.
- **Failure modes:** `cpu_count` and `disk_mb` are **secondary** anti-DoS controls — when the host lacks the affordance (`taskset` absent under the namespace tiers; the writable layer reports it can't be size-capped via the `diskQuotaSupported` check, on any tier), exec-sandbox **omits that one cap, prints a `WARNING` to stderr naming the control, records it in `sandbox_status.limits.degraded`, and continues** (the run is not failed — agent-builder ADR 027). Under **Firecracker `cpu_count` is *not* a secondary control** — it is the load-bearing `vcpu_count`, never degraded (there is no `taskset` to be absent). `memory_mb`/`pids` are load-bearing: an inability to apply them (e.g. `prlimit` absent under bubblewrap) is returned as `{error: …}`, not silently dropped.
- **References:** ADR 003 (cpu/mem/pids/disk/timeout) + ADR 007 (output cap above the tier seam) + ADR 010 D4 (the Firecracker mapping: cpu→vcpu_count, mem→mem_size_mib, disk→writable-drive size, pids→in-guest RLIMIT_NPROC; timeout/output host-side); agent-builder ADR 027 (degrade) / ADR 028 (runtime-aware verification); `limits.go` (`parseLimits` / `capWriter`); `run.go` `Run` (timeout/kill + host capture cap) / `bubblewrapBackend` / `bwrapArgv` / `outputTruncated` / `limitsReport`; `gvisor.go` `gvisorBackend` / `applyLimitsToOCISpec`; `firecracker.go` `firecrackerConfig` (vcpu/mem/pids-cmdline mapping) / `buildPayloadDrive` / `payloadDriveSizeMB` / `firecrackerBackend.Argv` (disk degrade, no taskset); `guest/rootfs/init/init` (in-guest `ulimit -u` + `setpriv` privilege drop); tests `TestParseLimits`, `TestParseLimits_MaxOutputBytes`, `TestTimeoutTerminatesPayload`, `TestMemoryLimitKillsPayload_Bwrap`, `TestPidsLimitRejectsForkBomb_Bwrap`, `TestDiskLimitBlocksWrites_Bwrap`, `TestCPUAffinity_Bwrap`, `TestDiskQuotaDegradesGracefully_Bwrap`, `TestGvisorEnforcesLimits`, `TestGvisorOCISpecCarriesLimits`, `TestCapWriter`, `TestOutputTruncatedRecord`, `TestOutputCapTruncates_Bwrap`, `TestOutputCapTruncates_Gvisor`, `TestOutputCapDoesNotTouchBwrapArgv`, `TestNoOutputCapFullOutput_Bwrap`, `TestFirecrackerCPUCountMapsToVcpuCount`, `TestFirecrackerMemoryMapsToMemSizeMib`, `TestFirecrackerDiskMapsToDriveSize`, `TestFirecrackerPidsMapsToInGuestNproc`, `TestFirecrackerZeroLimitsLeaveDefaults`, `TestFirecrackerDiskDegradesWhenUnsizeable`, `TestFirecrackerTimeoutNotInConfig`, `TestFirecrackerOutputCapNotInConfig`, `TestFirecrackerMemoryLimitOOMs_E2E`, `TestFirecrackerPidsForkBomb_E2E`, `TestFirecrackerVcpuObservable_E2E`.

### B-010: Mount a writable host working directory at /work

- **Trigger:** a run carries a non-empty `run.workdir` (a host path).
- **Response:** the path is validated before spawn (`validateWorkdir`: trimmed, made absolute, must be an existing **directory**) and then bind-mounted **read-write** at `/work` inside the sandbox, with the payload's cwd set to `/work`. Under bubblewrap this is `--bind <workdir> /work --chdir /work` (writable — **not** `--ro-bind`); under gVisor it is an OCI `/work` bind mount whose `options` omit `ro` plus `process.cwd = "/work"` (`applyWorkdirToOCISpec`). A file seeded in the host dir is readable by the payload at `/work`, and a file the payload writes under `/work` persists to the host directory after the run. `/work` is the **only** writable host surface: the rootfs and system dirs (`/usr`, `/etc`, …) stay read-only and the network namespace stays unshared (the no-network invariant is untouched — the only egress is still `/proxy.sock`).
- **Side effects:** the spawn argv (bwrap) and/or OCI `config.json` (gVisor) carry the writable `/work` mount and the cwd; the payload's writes land in the caller-supplied host directory.
- **Failure modes:** a non-empty `run.workdir` that does not exist, or resolves to a non-directory, returns `{error: "invalid run.workdir: …"}` **before** the proxy starts or any payload runs — there is no silent fall-back to a no-mount run. An empty/absent `run.workdir` produces **no** `/work` mount and leaves cwd unchanged (backward compatible).
- **References:** ADR 004; agent-builder `containment/execution-box/run.sh` (`/work,rw` + `--workdir /work`) / `internal/sandbox/podman/run.go` `validateWorktree`; `run.go` `validateWorkdir` / `bwrapArgv`; `gvisor.go` `applyWorkdirToOCISpec`; tests `TestValidateWorkdir`, `TestWorkdirSeededFileReadable_Bwrap`, `TestWorkdirWritePersists_Bwrap`, `TestWorkdirIsCwd_Bwrap`, `TestNoWorkdirNoMount_Bwrap`, `TestBadWorkdirFailsLoud`, `TestWorkdirEndToEnd_Gvisor`, `TestWorkdirOCISpec`, `TestOnlyWorkdirWritable_Bwrap`.

### B-011: Mount read-only host paths (FileRead) and provision the payload's env/PATH

- **Trigger:** a run carries one or more `{"type":"FileRead","paths":[…]}` capabilities in `run.profile.capabilities`, and/or a non-empty `run.env`.
- **Response:** the FileRead paths are collected (`fileReadPaths`; multiple entries union their lists) and validated before spawn (`validateFileReads`: each must be **absolute** and **exist**) — then each is bind-mounted **read-only** at the **same** host path inside the sandbox. Under bubblewrap this is `--ro-bind <path> <path>` (read-only — **not** the writable `--bind`); under gVisor it is an OCI mount `{destination:<path>, type:bind, source:<path>, options:[ro,rbind]}` (`applyFileReadToOCISpec`). `run.env` is exported into the sandbox: a `PATH` entry replaces the bare default `PATH=/usr/bin:/bin`, every other entry is exported `k=v` (bwrap `--setenv`; gVisor `process.env` via `applyEnvToOCISpec`), in a deterministic order (PATH first, then sorted keys). A payload can **read and execute** a FileRead-mounted tool, and — with the tool's dir on `run.env["PATH"]` — resolve it by name (`command -v <tool>`). A FileRead mount is **read-only**: a write to it fails (EROFS/permission), distinct from the writable `/work` (B-010). FileRead adds read-only host paths and PATH/env entries only — it opens **no** egress and **no** writable surface; the network namespace stays unshared (the only egress is still `/proxy.sock`).
- **Side effects:** the spawn argv (bwrap) and/or OCI `config.json` (gVisor) carry the read-only FileRead mounts and the provisioned env.
- **Failure modes:** a FileRead path that is **relative** or does **not exist** returns `{error: "invalid FileRead path: …"}` **before** the proxy starts or any payload runs — there is no silent fall-back to a no-mount run (same ordering as `validateWorkdir`). An empty/absent FileRead and an empty/absent `run.env` produce **no** extra mounts and the bare default PATH (backward compatible).
- **References:** ADR 005; agent-builder `containment/execution-box/run.sh` (`--mount …,ro` gate-tools dir + `gate_tool_path` PATH; `resolve_gate_tools`); `run.go` `fileReadPaths` / `validateFileReads` / `bwrapArgv` / `envSetenvPairs`; `gvisor.go` `applyFileReadToOCISpec` / `applyEnvToOCISpec` / `envList`; tests `TestFileReadParsing`, `TestValidateFileReads`, `TestFileReadMountReadableExecutable_Bwrap`, `TestFileReadOnPathResolves_Bwrap`, `TestFileReadMountIsReadOnly_Bwrap`, `TestNoFileReadNoEnv_Bwrap`, `TestBadFileReadFailsLoud`, `TestFileReadArgv_Bwrap`, `TestFileReadEndToEnd_Gvisor`, `TestFileReadOCISpec`, `TestFileReadRegressionBaseUnchanged`.

### B-012: Build a pristine per-run baseline and restore it to a clean slate

- **Trigger:** every run (`Run()`) — the baseline is built before the payload executes; `restore` is the named reset operation a reuse path would call between invocations.
- **Response:** `Run()` builds a `sandboxBaseline` (`snapshotBaseline`) capturing the pristine per-run state — a fresh host work dir with `payload.sh` (mode 0600) seeded into it, the fresh per-run `/proxy.sock` path under that dir, and a fresh `EgressProxy` with an empty credential map. This is the **snapshot**: the named baseline the leak-proof property is asserted against. `restore` returns the sandbox to that baseline — it wipes the writable surface back to exactly the pristine file set (re-seeding only `payload.sh`; anything the payload wrote under the writable surface is removed) and clears the proxy credential map (subsuming `Wipe()`). After `restore`, the writable surface **and** the credential map are byte-for-byte equal to a freshly-built baseline. The default one-shot path is `snapshot → run → teardown` (terminal `os.RemoveAll(work)` + `proxy.Wipe()`), **observationally identical** to the prior inlined setup/teardown — no result-schema change, no argv/OCI-spec change. The boundary is **host-side only and tier-independent** (ADR 009): it covers the host work dir, `payload.sh`, and the host-side proxy credential map — the same state under bubblewrap and gVisor — and does **not** reach inside a tier's kernel root.
- **Side effects:** the work dir + `payload.sh` are created at snapshot and removed at teardown (one-shot), or reset to the pristine file set at `restore` (reuse); the proxy credential map is cleared by `restore`/`Wipe`. A restored sandbox rebuilds its spawn argv/OCI spec from the same `bwrapArgv`/`gvisorOCISpec` path a fresh run uses — `--unshare-all` (or the path-less OCI netns), no `--share-net`, and the **same fresh** `/proxy.sock` (never a stale socket); `restore` re-binds no stale credential. The no-network + proxy-only-egress invariant is untouched and the writable surface never widens.
- **Failure modes:** a temp-dir or seed failure at snapshot returns `{error: "baseline prepare failed: <err>"}` before the proxy starts or any payload runs (no silent fall-back). `restore` on an already-pristine baseline is a no-op (no error). No long-lived process is introduced and no reuse trigger exists yet — the mechanism is proven leak-free; wiring a second invocation onto a restored baseline is a deferred follow-on (ADR 009 Q2/Q3).
- **References:** ADR 009 (host-side reset; open-question dispositions; reopening condition for the warm-pool variant); `snapshot.go` `sandboxBaseline` / `snapshotBaseline` / `restore` / `teardown` / `writableSurface` / `credentialHosts`; `run.go` `Run`; `proxy.go` `Wipe`; tests `TestSnapshotCapturesPristineBaseline`, `TestRestoreReturnsToBaseline`, `TestNoStateLeaksAcrossRestore`, `TestRestoredSandboxKeepsNoNetworkInvariant`, `TestNoCredentialLeaksAcrossRestore`, `TestSecondRunCannotSeeFirstRunFiles_Bwrap`, `TestOneShotRunUnchanged_Bwrap`.

### B-013: Apply a default-deny seccomp profile on Tier-1

- **Trigger:** every Tier-1 (bubblewrap) run reaches the spawn step (`bubblewrapBackend.Argv`). Tier-2 (gVisor) and Tier-3 (Firecracker) do **not** trigger this — they self-filter every syscall (ADR 016).
- **Response:** the backend loads a build-time-generated, sha256-pinned cBPF blob (`loadTier1Seccomp`: it verifies the embedded `seccomp/tier1.bpf` against the embedded `seccomp/tier1.bpf.sha256` with stdlib `crypto/sha256`, then materializes it as an open fd), threads that fd into the spawn as `cmd.ExtraFiles[0]` (child **fd 3**), and emits `--seccomp 3` on the bwrap argv. bwrap installs the program as a seccomp filter on the payload. The profile is **default-deny + allowlist** (`SCMP_ACT_ERRNO(EPERM)` default): the common-case syscalls a payload shell + the proxy client (`curl` over the Unix socket) need stay allowed, while the dangerous family — `keyctl`, `add_key`, `request_key`, `ptrace`, `process_vm_readv`/`writev`, `userfaultfd`, `bpf`, `perf_event_open`, the `mount`/`umount2`/`pivot_root`, `kexec_load`/`kexec_file_load`, and `init_module`/`finit_module`/`delete_module` families — returns `EPERM`. The filter **adds to** the no-network model: `--unshare-all` is kept and `--share-net` stays absent. When an env-mode credential is also delivered (B-003 / ADR 015), the `--args` pipe follows the seccomp file as `ExtraFiles[1]` (child **fd 4**), so `--args 4` and `--seccomp 3` coexist.
- **Side effects:** the spawn argv carries `--seccomp 3`; the verified cBPF blob is materialized to an unlinked temp file whose open fd is inherited by the child and closed after spawn (nothing persists on disk). The policy source of truth is `seccomp/tier1-policy.json`; the committed blob + pin are `seccomp/tier1.bpf` and `seccomp/tier1.bpf.sha256`.
- **Failure modes:** if the blob fails to verify against the pin (tampered, stale, or truncated), `loadTier1Seccomp` returns a hard error and the backend **fails the run** — it **never** falls back to spawning bwrap **without** `--seccomp`. This is fail-fast: an unverifiable kernel-attack-surface filter aborts the run, it does not silently weaken it.
- **References:** ADR 016; Docker/podman default seccomp profile (model); `seccomp.go` `loadTier1Seccomp` / `verifyTier1SeccompBytes` / `parseSeccompPin`; `run.go` `bubblewrapBackend.Argv` / `bwrapArgv`; `seccomp/tier1-policy.json` / `seccomp/build.sh` / `seccomp/gen.c`; tests `TestBwrapArgvCarriesSeccomp`, `TestBubblewrapBackendThreadsSeccompFD`, `TestSeccompLoaderAcceptsMatchingBlob`, `TestSeccompLoaderFailsFastOnMismatch`, `TestBackendPropagatesSeccompLoadFailure`, `TestPolicyDenySetCoversDangerousFamily`, `TestSeccompBlobReproducibleFromPolicy`, `TestKeyctlBlockedReturnsEPERM_Bwrap`, `TestCommonCasePayloadStillRuns_Bwrap`, `TestFitnessTier1SeccompPositive`, `TestFitnessTier1SeccompNegative`.

### B-014: Route Tier-3 (Firecracker) microVM egress over a vsock bridge with no NIC

- **Trigger:** a Tier-3 (Firecracker) payload makes an HTTP request through the in-guest `/proxy.sock`.
- **Response — the microVM egress flow.** A microVM has neither `--unshare-all` nor an OCI netns, and it cannot bind-mount the host `/proxy.sock` (separate kernel, separate VFS). The two load-bearing invariants are re-expressed in microVM terms (ADR 010 D2): **(1) No NIC, ever.** The generated firecracker config issues **no** `network-interfaces` body, so the guest has **no network device at all** — no TAP, no bridge, no route. "No network" is true **by omission** — the microVM analogue of `bwrap --unshare-all`; `configHasNoNIC` re-asserts this on the wired config (`wiredFirecrackerConfig`) and the no-NIC guard rejects any config that carries a `network-interface(s)` key. **(2) The proxy is reached over a virtio-vsock device, not a bind-mount.** The firecracker config wires a `vsock` device with a host-side `uds_path`; the host side of that vsock terminates at the **host bridge** (`vsockBridge`, `startVsockBridge`), which forwards every byte to the live `EgressProxy`'s Unix socket. The proxy (`proxy.go`) is reached **as-is** over this transport — its domain allowlist, per-host verb check, and host-side credential injection are byte-for-byte unchanged; the vsock is a **transport substitution** for the bind-mount, not a change to the trust boundary. **(3) A dumb guest-side shim presents `/proxy.sock` to the payload.** The guest-side shim (`vsockshim.go`, `startGuestShim`) listens on `/proxy.sock` inside the guest and pumps bytes — bidirectionally and byte-exactly — over the vsock to the host bridge. It is a **byte pump only**: it imports no HTTP package, holds no host set or method map, and stores no secret, so it makes no egress decision and cannot leak a secret it never holds. The block decision (`403 blocked-by-allowlist`) is made host-side by the `EgressProxy`, never in the guest. The payload's contract is identical across tiers — it always talks to a Unix socket at `/proxy.sock`. **(4) The credential never enters the guest.** The proxy injects the credential header host-side, **after** the request crosses the vsock back to the host — identical to the bind-mount case. The credential value appears in **none** of: the guest env, the guest args, the guest stdout, the host spawn argv, or the returned `result["stdout"]` (`assertCredNotInGuest` over `guestSurfaces`, the microVM extension of the task-009 host surface set).
- **Q4 (ADR 010) — shim location and lifecycle (resolved).** The guest-side `/proxy.sock` shim **ships baked into the read-only rootfs base** as part of the trusted computing base (the pinned RO Alpine base carries only `/sbin/init` plus this forwarder) and is **started by `/sbin/init` at boot**. Its dumbness is kept **by construction** — it is a bidirectional byte pump — and is an **auditable property**: the shim source imports no HTTP package and contains no host set, no method map, and no secret handling, asserted by a source-level test (`TestShimHasNoHTTPNoCredNoAllowlist`).
- **Side effects:** the host bridge dials the live proxy socket per accepted vsock connection and relays bytes; no host network device, route, or NIC is ever created. A blocked host produces no outbound request (the host-side proxy decides). No credential value crosses the vsock into the guest.
- **Failure modes:** if the host side of the vsock (the bridge) is unreachable, the shim drops the connection — there is **no second route** out of the guest (fail closed: no bridge → no egress). A direct (NIC-less) network attempt from the guest fails because there is no network device. The end-to-end allow/block/credential behavior is exercised on a booted guest under `/dev/kvm` (task 015's launch path); the bridge wiring, the dumb-shim pump, the no-NIC guard, and the guest-surface leak-scan run without a guest.
- **References:** ADR 010 D2 + Q4 resolution note; `vsockbridge.go` `startVsockBridge` / `vsockBridge` / `wiredFirecrackerConfig` / `assertCredNotInGuest` / `guestSurfaces`; `vsockshim.go` `startGuestShim` / `guestShim`; `firecracker.go` `firecrackerConfig` / `configHasNoNIC`; `proxy.go` `EgressProxy` (unchanged); tests `TestVsockBridgeWiresToLiveEgressProxy`, `TestProxyGoUnchangedByVsockTask`, `TestWiredFirecrackerConfigHasNoNIC`, `TestWiredConfigNoNICGuardRejectsNIC`, `TestShimIsTransparentBidirectionalPump`, `TestShimHasNoHTTPNoCredNoAllowlist`, `TestCredNotInGuestDetectorRejectsLeak`, `TestQ4ResolvedAndRecorded` (and the guest-boot-gated `TestCredentialNeverReachesGuest_E2E` / `TestGuestReachesAllowlistedHost_E2E` / `TestGuestBlockedNonAllowlistedAndDirectNetFails_E2E`, which run on task 015's boot path).

### B-015: Boot the Tier-3 (Firecracker) guest and run the payload (no jailer)

- **Trigger:** a run reaches the spawn step with `req.run.tier == "firecracker"`.
- **Response — the microVM boot flow (ADR 010 D1/D3 + Amendment 1).** `firecrackerBackend.Argv`:
  **(1) Verifies the pinned guest artifacts.** A stdlib `crypto/sha256` loader (`loadGuestArtifacts`) resolves the vendored, pinned guest kernel (`guest/kernel/vmlinux-6.1.176`, an uncompressed x86_64 ELF) and rootfs (`guest/rootfs/base.ext4`, a read-only Alpine ext4) and checks **both** against their sibling `.sha256` pins. Any mismatch / missing pin **fails fast** (the pin is the supply-chain control; A1.Q1) — the backend never boots an unverified artifact.
  **(2) Builds a per-run bundle.** A temp bundle holds the generated `vm-config.json` and a small **writable payload drive** (`payload.ext4`, built with `mkfs.ext4 -d`) carrying `payload.sh`. The untrusted payload is **never baked into the read-only base** — it rides on this separate drive (`/dev/vdb` in the guest), keeping the base digest stable.
  **(3) Starts the host-side vsock egress bridge** (B-014) on `<bundle>/proxy.vsock_<port>`, fronting the live `EgressProxy`.
  **(4) Launches `firecracker` directly under the unprivileged wrapper — NO jailer (A1.Q3).** The spawn argv is `bwrap --unshare-all --die-with-parent --uid 65534 --gid 65534 [read-only system + kernel + rootfs binds] --dev-bind /dev/kvm /dev/kvm --bind <bundle> <bundle> -- exec-sandbox fc-launch <bundle>`. `bwrap --unshare-all` creates a fresh user namespace, and `--uid 65534 --gid 65534` run the firecracker child as an unprivileged `nobody` **inside** it — the no-jailer analogue of the jailer's drop to an unprivileged uid. Together with `limits.go` (rlimit/affinity caps) this supplies the chroot/`pivot_root` + mnt/pid/ipc/net/user/uts/cgroup namespaces + non-host uid the jailer would otherwise provide; firecracker self-installs its seccomp filters regardless. `/dev/kvm` is the **only** device exposed; no root, no setuid, no caps. There is **no `--share-net`** (the no-network invariant) and **no `ip=`** kernel arg (no NIC to configure).
  **(5) The in-bwrap launcher (`fc-launch`) drives the REST API in order** over the firecracker API Unix socket — `PUT /machine-config` → `/boot-source` → `/drives/...` → `/vsock` → `PUT /actions {InstanceStart}` — and **never** `PUT /network-interfaces` (no NIC at the API level). Boot args are `console=ttyS0 reboot=k panic=1` (plus `pci=off` + i8042 quirks, and — when FileRead paths are present — an `exec_sandbox.fileread=<dev>:<path>,…` mapping, B-016). The guest `/sbin/init` starts the vsock shim, mounts `/dev/vdb` (payload) read-only, mounts `/dev/vdc` (the writable `/work` surface) read-write and each FileRead drive (`/dev/vdd…`) read-only per B-016, cd's into `/work`, and runs `/usr/bin/sh /payload.sh` — the same entry point as every other tier.
  **(6) Exit + output flow through the UNCHANGED host capture path.** The guest emits its exit code on the serial console as a `__EXEC_SANDBOX_EXIT__ N` sentinel; `fc-launch` strips kernel/firecracker log chatter, forwards the payload's stdout, and **exits with the guest's exit code**. Because `fc-launch`'s process exit code *is* the guest's, `Run()`'s host-side `capWriter` capture, process-group SIGKILL on the wall-clock deadline, and exit-code mapping apply unchanged: clean exit → guest code; non-zero → guest code; `timeout_sec` exceeded → the whole group (fc-launch + firecracker) is SIGKILLed → `status: "timeout"`, `exit_code 137`.
  **(7) The live firecracker child's constraints are ≥ jailer, observed host-side (A1.Q3, ADR 010 Amendment 2).** Read straight from `/proc/<pid>/*` on the running child (matched by its actual `exe` basename, not a spoofable cmdline substring): **all 7 namespaces** (net/user/mnt/pid/ipc/uts/cgroup) differ from an unwrapped host process; the **user-namespace `uid_map` is `"65534 <hostuid> 1"`** — the in-namespace uid (65534) is a non-host uid (the `/proc/<pid>/status` Uid line still reports the *owning* host uid because the kernel translates namespaced ids for an outside observer, so the `uid_map`, not the status Uid, is the load-bearing observable); **`CapEff` is all-zero** + **`NoNewPrivs` is 1** (privilege dropped, unregainable); the child's **root mount is a `pivot_root` tmpfs newroot** (`0:NNN /newroot`, fs `tmpfs`), not the host's real block-device `/`. Resource limits are enforced by rlimits/affinity (`limits.go`, ADR 003) layered above bwrap — not per-run cgroup placement — and the cgroup *namespace* is unshared. The assertion **bites**: a wrapper missing `--unshare-all`, sharing any namespace with the host, omitting `--uid`, regaining capability, or omitting the `pivot_root` fails a distinct check.
- **Side effects:** a per-run bundle dir + payload drive are created and removed (cleanup func); a host vsock bridge is started and stopped (same cleanup); one `firecracker` child boots one microVM and is terminated at teardown (`reboot=k` on a clean run, SIGKILL on timeout) — no guest, bundle, or socket outlives the run.
- **Failure modes:** a missing/unverifiable pinned kernel or rootfs (sha256 mismatch), an absent `firecracker` binary, or an inaccessible `/dev/kvm` is a spawn error (`exit_code 127`) surfaced through the host path — **never** a silent fall-back to a weaker tier (mirrors the gVisor `runsc`-absent behavior). A guest that panics before emitting the exit sentinel surfaces `127`, not a fake success.
- **References:** ADR 010 D1/D3 + Amendment 1 (A1.Q1, A1.Q3); `firecracker.go` `firecrackerBackend.Argv` / `firecrackerArgv` / `buildPayloadDrive` / `addPayloadDrive`; `fclaunch.go` `fcLaunchMain` / `firecrackerBootSequence` / `driveFirecrackerAPI` / `streamConsole`; `fcartifacts.go` `loadGuestArtifacts` / `verifyPinnedDigest`; `guest/kernel/`, `guest/rootfs/` (vendored pinned artifacts + `build.sh`); `run.go` `Run` (host capture, unchanged); tests `TestFirecrackerGuestBoot_E2E`, `TestFirecrackerCleanExit_E2E`, `TestFirecrackerNonZeroExit_E2E`, `TestFirecrackerTimeout_E2E`, `TestFirecrackerConstraintsGEJailer_Live` / `_Argv`, `TestFirecrackerRESTOrderNoNIC`, `TestGuestArtifactsVerifyAndFailFast`, `TestGuestKernelProvenanceVendored`, `TestFirecrackerMissingArtifactsIsHardError`, `TestFirecrackerRunMissingPrereqIsExit127`, `TestFirecrackerArgvIsBwrapDirectNoJailer`, `TestFirecrackerBundleCreatedAndTornDown`.

### B-016: Present `/work` (writable) and FileRead paths (read-only) to the Tier-3 microVM as block devices

- **Trigger:** a `firecracker`-tier run reaches the spawn step with a non-empty `run.workdir` and/or one or more `FileRead{paths}` (the same validated inputs as B-010/B-011, which are tier-independent and unchanged).
- **Response — the microVM mount flow (ADR 010 D3 / Q2 resolution: block device).** A microVM has no host bind-mount (separate kernel, separate VFS), so the writable `/work` and the read-only FileRead paths reach the guest as **block-device drives**, the same mechanism as the rootfs (B-015). The choice is recorded as the **Q2 resolution**: virtio-fs is unavailable (no `virtiofsd` on the host), block devices give the cleanest isolation boundary, and the host tooling supports an **unprivileged** implementation end to end.
  **(1) `/work` is a writable ext4 drive (`/dev/vdc`, `is_read_only:false`).** `firecrackerBackend.Argv` seeds a per-run ext4 image from `run.workdir` via `mkfs.ext4 -d <workdir>` (**copy-in**, no root/loop-mount), so a host-seeded `/work` file is readable in-guest. The guest init mounts it read-write at `/work` and the payload's cwd is `/work`, so a bare relative write lands on the writable surface. After the guest exits, the backend `cleanup` func copies the guest's writes back into `run.workdir` via `debugfs -R "rdump / <hostdir>"` (**copy-out**, userspace, no root/loop-mount). **Semantic nuance:** writes therefore persist to the host work dir **at teardown**, not live — ADR 004's "writes persist to the host work dir" holds with persist-at-teardown timing under the block-device mechanism.
  **(2) Each FileRead path is a read-only ext4 drive (`/dev/vdd`, `/dev/vde`, …, `is_read_only:true`).** Each is seeded from the host path (copy-in) and exposed by the guest init read-only at the **same** absolute host path the payload would use (mirroring bwrap's `--ro-bind <p> <p>`); the device→mountpoint mapping rides the **trusted kernel cmdline** (`exec_sandbox.fileread=<dev>:<path>,…`), not the untrusted payload drive. A guest write fails (the drive is read-only at the virtio layer) and the host file is **never** modified or created — there is **no copy-out** for FileRead drives (ADR 005 / F-007). The copy-in is a snapshot, so a guest cannot reach the host file through this path.
  **(3) `/work` is the single writable surface (F-006).** The rootfs (`/dev/vda`), payload (`/dev/vdb`), and every FileRead drive are `is_read_only:true`; only the `/work` drive is `is_read_only:false`. A **config-level guard** (`validateDriveReadOnly`) asserts **exactly one** writable drive (the `/work` surface, `drive_id "work"`) and refuses any writable non-`/work` drive or any writable FileRead drive **before** the config is written or booted — the negative case (a FileRead drive accidentally made writable) is a hard spawn error, proving the guard is not a no-op.
  **(4) The mount mechanism opens no network path.** The drives are block devices; the mount-wired config still carries **no** `network-interface(s)` key (`configHasNoNIC` re-asserted), and the vsock stays the only host↔guest channel besides the drives (the no-NIC invariant of B-014/B-015 is untouched).
- **Side effects:** a writable `work.ext4` and one `fileread-N.ext4` per FileRead path are built in the per-run bundle and removed with it; the `/work` image is read back via `debugfs` at teardown and the recovered files (excluding ext4 `lost+found`) are synced into `run.workdir`. No FileRead image is ever copied out.
- **Failure modes:** a bad `run.workdir` or FileRead path is a hard `{error}` **before** any side effect (B-010/B-011 validation, tier-independent and unchanged — the firecracker tier does not silently run unmounted). A `mkfs.ext4` absence or seed failure is a spawn error (the guest cannot get its `/work`/FileRead surface). A copy-out failure at teardown is best-effort (the host workdir keeps its pre-run state) and never aborts teardown — the run's stdout/exit_code are already captured. An empty `run.workdir` adds **no** `/work` drive (byte-for-byte the task-015 boot shape).
- **References:** ADR 010 D3 + Q2 resolution note; ADR 004 (writable `/work`, persist-at-teardown under the block device) + ADR 005 (read-only FileRead); `firecracker.go` `firecrackerBackend.Argv` / `addWorkdirDrive` / `addFileReadDrives` / `fileReadDriveID` / `fileReadGuestDev` / `validateDriveReadOnly` / `buildWorkdirDrive` / `buildFileReadDrive` / `copyOutWorkdir`; `guest/rootfs/init/init` (mount `/dev/vdc` rw at `/work`, cd `/work`; mount each FileRead drive ro at its host path) + `guest/rootfs/build.sh` (`/work` mountpoint in the base); `run.go` `validateWorkdir` / `validateFileReads` (unchanged); tests `TestFirecrackerWork_HostSeededFileReadableInGuest_E2E`, `TestFirecrackerWork_WritePersistsToHost_E2E`, `TestFirecrackerWork_OnlyWorkWritable_E2E`, `TestFirecrackerFileRead_ReadOnlyHostUnchanged_E2E`, `TestFirecrackerMount_OnlyWorkIsWritable_Config`, `TestFirecrackerMount_ReadOnlyGuardBites`, `TestFirecrackerMount_SecondWritableDriveRejected`, `TestFirecrackerMount_NoNIC`, `TestFirecrackerMount_DriveOrderAndFlags`, `TestFirecrackerMount_BadPathFailsBeforeSideEffects`.

## Edge cases and error behaviors

### B-006: Reject invalid invocation or unparseable request

- **Trigger:** the binary is invoked without the `run` subcommand, or stdin is not a valid `RunRequest` JSON, or stdin cannot be read.
- **Response:** missing/unknown subcommand prints `usage: exec-sandbox run …` to stderr and exits `2`. A stdin read error or a JSON parse error prints the error to stderr and exits `1`.
- **Side effects:** none (no sandbox is started, no audit emitted).
- **Failure modes:** this *is* the failure path.
- **References:** `main.go`.

### B-007: Fail fast if the egress proxy cannot start

- **Trigger:** the egress proxy fails to bind its Unix socket.
- **Response:** `Run` returns `{error: "proxy start failed: <err>"}` and does not run the payload.
- **Side effects:** the `spawn` audit event and any credential injection have already happened. A terminal audit event is emitted immediately before returning the `{error}`: `{actor:"exec-sandbox", action:"exit", target:sandbox_id, decision:"deny", context:{status:"proxy_start_failed", error:<msg>, request_id}}`. This closes the spawn-without-exit audit-completeness gap (ADR 013): every run that has emitted `spawn` emits a matching terminal event. Emission is best-effort — an empty `audit_socket` is a no-op; an IPC error is swallowed.
- **Failure modes:** this is the failure path. The payload does not run; no backend (`bwrap`/`runsc`) is selected or executed; no success result shape is returned.
- **References:** ADR 013; `run.go` `Run`; `proxy.go` `Start`.

## Behavioral invariants

- **The sandbox never has network access** regardless of profile, tier, or secret_refs. The only egress is the proxy socket — bind-mounted as `/proxy.sock` under bubblewrap/gVisor, presented in the guest by the dumb vsock shim under Firecracker (B-014). The no-network invariant has three enforcement points: `bwrap --unshare-all` (Tier-1), the empty OCI netns + `runsc --network=none` (Tier-2), and the **absence of any `network-interface` in the firecracker config** plus the vsock-bridged proxy (Tier-3, B-014).
- **A proxy-mode credential value is never observable from inside the sandbox** — not in env, args, payload, or stdout. It exists only on the host-side proxy. Under Firecracker the credential is injected host-side **after** the vsock hop, so it never crosses into the guest (B-014): it appears in none of the guest env/args/stdout, the host spawn argv, or the returned stdout.
- **Audit and vault calls are best-effort and non-fatal** except proxy-start failure (B-007), which aborts the run before any payload executes. A run that has emitted `spawn` always emits a matching terminal event — either `exit`/allow on success, or `exit`/deny with `status:"proxy_start_failed"` on early proxy failure (ADR 013).
- **Every requested `profile.limits` cap is enforced or its degradation is recorded** (B-009): a cap is applied on the active backend, or — for the secondary controls only (`cpu_count`/`disk_mb` under the namespace tiers; `disk_mb` under Firecracker — `cpu_count` is a load-bearing `vcpu_count` there, never degraded) — it appears in `sandbox_status.limits.degraded` with a stderr `WARNING`. No requested cap is ever silently ignored. `max_output_bytes` is enforced host-side **above** the `tier` seam, identically under every backend (no per-tier wiring, no degrade path); truncation is never silent — the capped streams are listed in `sandbox_status.limits.output_truncated`.
- **The only writable host surface is `run.workdir` at `/work`** (B-010), and only when the caller sets it. The rootfs, system dirs, `/payload.sh`, and any `FileRead{paths}` mounts (B-011) are always read-only; the writable surface never widens beyond the single caller-supplied directory, and no host mount (writable or read-only) ever opens an egress path (the network stays unshared).
- **Every successful run returns the full result shape** (B-005); there is no partial-result path on the success side.
- **A restored sandbox is indistinguishable from a freshly-built one** (B-012): after `restore`, the writable surface and the proxy credential map are byte-for-byte equal to a fresh baseline — no file, env var, or credential from a prior run survives, the netns stays unshared, and the only egress is the same fresh `/proxy.sock` (no stale socket, no re-bound stale credential). The one-shot path (`snapshot → run → teardown`) is observationally identical to the prior inlined behavior. The reset is host-side only and tier-independent (ADR 009).
